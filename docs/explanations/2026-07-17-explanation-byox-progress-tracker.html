Turning a static ladder into a synced progress tracker

Change explanation · build-your-x @ ff4cf9f · 2026-07-17

The build-your-own-x repo used to be a read-only wall of ~359 tutorials sorted by difficulty. This change turns it into something you work through: a byox command to mark guides done and note where you built them, a dashboard that shows your progress, and cross-device sync so your progress follows you between machines.

1 · Background

The repo started life as a tiny static-site generator. One script, parse_guides.py, reads a copy of the upstream build-your-own-x README and emits guides.json — a list of ~359 tutorials, each with a title, url, tier (beginner…expert) and a difficulty score. A second script, build_pages.py, injects that JSON into an HTML template and writes guide-ladder.html: a single offline page you open with file://, with search and tier/language filters.

Crucially, that page was stateless. There was no notion of "I finished this one." Nowhere did the code record completion, and nothing stored what you built. If you wanted to track progress you did it in your head.

Two facts about the old code matter for the change:
  • parse_guides.parse_readme() ends with guides.sort(key=lambda g:(g.score, g.title.lower())) — the list is re-sorted every rebuild. So a guide's position in the list is not stable.
  • build_pages.build_guide_page() works by string-replacing a marker (/*__GUIDES__*/) in the template with const GUIDES = [...]. That marker pattern is the hook the new progress data rides in on.

The change also lifts the whole package out of a monorepo into its own repo and, alongside the generator, adds the builds themselves under builds/ — but the interesting engineering is the tracker, so that's what the rest of this doc is about.

2 · Intuition

Two ideas carry the whole feature. Get these and the code reads itself.

Idea 1 — progress is a tiny key–value store, keyed by URL

Completion state lives in one JSON file at ~/.local/share/byox_ladder/progress.json, mapping a guide to {done, note, updated_ms}. The subtle choice is the key. Not the title, not the row number — the URL.

Why? Remember the list is re-sorted (score, title) on every rebuild, and upstream titles get edited. Key on either and your "done" marks would silently drift onto the wrong guide after a rebuild. The URL is the one field that stays put.

progress.json
{
 "https://citw.dev/…/create-your-own-cli-tool": {
  "done": true,
  "note": "builds/tool-tutorial",
  "updated_ms": 1784310000000
 }
}
→
dashboard row
✓ Create a CLI tool in Javascript builds/tool-tutorial
only done guides are baked in

At build time, build_pages reads that store and injects a second payload, const PROGRESS = {url: {done, note}}, next to GUIDES. The template's JavaScript looks each guide up by URL and draws a ✓, a green row, the build-note, and per-tier progress bars. The page stays a pure offline file — the state is baked in, not stored in the browser.

Idea 2 — sync is a per-device log that merges by "latest wins"

To follow you across machines, progress rides on the shared crdt_sync library. There's no server: each device writes its own file to a private GitHub repo at byox-ladder/devices/<device>/log.json, and on each sync pulls the other devices' files and merges them.

Merge is last-writer-wins per field, ordered by a hybrid-logical-clock timestamp seeded from updated_ms. If your laptop marked a guide done at t=300 and your desktop noted it at t=100, the t=300 edit wins; distinct guides from each device simply union together.

After a sync each side holds both guides. Play with the exact rule below.

3 · Code

Grouped by concern rather than by file. Five new modules plus edits to the builder and template.

a · The progress store — _progress.py

A dataclass plus load/save. Two details earn their keep: the URL key, and an atomic write (temp file then replace) so a crash mid-write can't truncate your progress. Reads tolerate a missing or corrupt file by returning an empty store — losing local progress must never crash the CLI.

# key = guide URL (the only rebuild-stable id)
def mark(progress, url, *, done=None, note=None, when_ms=None):
    entry = progress.get(url) or ProgressEntry(url, False, "", stamp)
    if done is not None: entry.done = done      # only touch fields passed
    if note is not None: entry.note = note
    entry.updated_ms = stamp                    # bump so this edit wins merges

b · Naming a guide loosely — _resolve.py

You don't want to paste a URL every time. resolve() takes a loose token and tries, in order of specificity: exact URL → URL/slug substring → title substring. The first level that matches decides — and if several match at that level, it raises with the candidate list instead of silently guessing.

picked = _pick(url_matches, query)   # 1 → return, >1 → raise w/ candidates
if picked is not None: return picked
title_matches = [g for g in guides if lowered in g["title"].lower()]

# byox done "cli-tool" → unique, marks it

c · The CLI — _cli.py + __main__.py

An argparse dispatcher for done / undone / note / status / build / sync. Output goes through a small _emit() writer (not print) so tests can capture it and the strict linter stays happy. A single except GuideResolutionError at the top turns any resolution problem into a friendly message plus the candidate list.

d · Baking progress into the page — build_pages.py + template

The builder gains a second marker. It now reads the store, keeps only done guides, and injects them beside the guides list. The template's JS looks each guide up by URL to draw the ✓, note, per-tier bars, and a "hide completed" filter.

  for marker in (MARKER, PROGRESS_MARKER):
      if marker not in template:
          raise ValueError(...)                # both must exist
  page = template.replace(MARKER, f"const GUIDES = {inline(guides)};")
  page = page.replace(PROGRESS_MARKER, f"const PROGRESS = {inline(done_only)};")

A subtle bug fixed during review: the build-note badge first reused the CSS class .note, which the header's explanatory paragraph also used. The badge's white-space:nowrap then leaked onto that paragraph and made it overflow off-screen. Renaming the badge to .buildnote fixed it — a reminder that global CSS class names are a shared namespace.

e · Sync — _sync.py (+ units, install.sh)

The adapters translate the store to and from crdt_sync records (progress_to_log / log_to_progress), stamping each field with an HLC seeded from updated_ms. run_sync() reads the token, and if it's absent returns a benign unconfigured outcome — sync is optional, the tool works fully offline. A real network error is caught and reported, never raised past this boundary.

  token = read_token(token_file)
  if token is None:
      return SyncOutcome("unconfigured", 0, 0, str(token_file))  # offline no-op
  try:
      merged = sync_log(client=…, device_id=node, path_prefix=PREFIX, …)
  except GitHubSyncError as exc:
      return SyncOutcome("error", 0, 0, str(exc))               # logged, not raised
The gotcha worth remembering: a device never pulls its own remote file. So wiping local progress.json and syncing on the same machine overwrites its remote with the empty state. That's fine in normal use — the local store is the source of truth — but it means "restore from the cloud" only works from a different device.

Backing it all: 71 tests at 100% branch coverage, a network-blocking test fixture so no test hits real GitHub, and a systemd timer that runs byox sync every 15 minutes.
